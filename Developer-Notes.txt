Should AST nodes should carry source spans? Yes but define my own wrapper to avoid coupling all AST code to idris2-parser:

public export
record SourceSpan where
  constructor MkSourceSpan
  startLine : Nat
  startCol  : Nat
  endLine   : Nat
  endCol    : Nat

public export
record Located a where
  constructor MkLocated
  span  : SourceSpan
  value : a

Parser.idr converts idris2-parser Bounded values into Leaf Located values.

Code pipeline:

Compiling Leaf source code
  -> Lexer
  -> Parser
  -> Surface AST
     - raw parsed syntax, source-shaped, names mostly textual
  -> Resolved AST (Name resolution)
     - same general shape, but names resolved to stable IDs
  -> Type/effect/linearity checker
  -> Typed AST
     - helps preserve the exact source-level structure after type checking 
  -> Typed High IR (WILL LIKELY SKIP)
    - proof-friendly core language, every term has checked type/effect/linearity information
  -> Low IR (Idris2 DSL) 
  -> Backend target: OpenQASM3/QIR

When a typed AST is useful:
- IDE features
- hover types
- refactoring
- source-to-source transformations
- better diagnostics
- linting
- format-aware analysis
- keeping qif/scratch/uncompute exactly source-shaped
